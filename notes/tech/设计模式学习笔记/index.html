    设计模式学习笔记

    课程来源：极客时间

    第一部分：面向对象基础

    面向对象四大特征

    • 封装（Encapsulation）：信息隐藏或数据访问保护

      • 类通过暴露有限的访问接口，授权外部仅能通过类提供的方式来访问内部信息或数据
      • 意义：类属性的安全可控；提高类的易用性，只暴露必要的操作

    • 抽象（Abstraction）：隐藏方法的具体实现

      • 调用者只关心功能，不需要知道功能的实现
      • 通过接口类或抽象类实现
      • 意义：调用客户端不需要面对所有的类属性

    • 继承（Inheritance）：代码复用

    • 多态（Polymorphism）：相同的方法调用在不同对象上产生不同的行为

      • duck type

    第二部分：设计原则与思想

    核心设计原则

    1. 单一职责原则（SRP）：一个类只负责一个职责
    2. 开闭原则（OCP）：对扩展开放，对修改关闭
    3. 里式替换原则（LSP）：子类可以替换父类
    4. 接口隔离原则（ISP）：接口隔离有三种应用
    5. 依赖倒置原则（DIP）：依赖倒置、控制反转、依赖注入
    6. KISS 原则：Keep It Simple and Stupid
    7. YAGNI 原则：You Aren’t Gonna Need It
    8. DRY 原则：Don’t Repeat Yourself
    9. 迪米特法则（LOD）：实现高内聚、松耦合

    重要设计思想

    组合优于继承

    • 组合（has-a）：旧类的对象作为新类的一部分，是一个属性变量或参数传递
    • 继承（is-a）：新类是旧类的一种，使用 extend 关键字

    第三部分：创建型设计模式

    单例模式（Singleton）

    解决的问题：一个类只允许创建一个对象（实例）

    应用场景：

    • 处理资源访问冲突（如多线程日志）
    • 表示全局唯一类（配置信息类、ID生成器）

    实现方式及对比：

    方式延迟加载线程安全性能备注
    饿汉式❌✅优加载类时创建实例
    懒汉式✅✅ 需加锁一般高并发下性能问题
    双重检测✅✅优需要 volatile 禁止指令重排
    静态内部类✅✅优JVM 保证唯一性和线程安全
    枚举✅✅优枚举特性保证唯一性

    单例模式的问题与替代方案：

    • 问题：对OOP不友好、隐藏依赖、不支持扩展、难以测试、不支持有参构造
    • 替代方案：静态类、依赖注入、工厂模式、IOC 容器

    分布式环境下的单例

    单例作用范围：进程唯一性（fork 子进程会复制所有内容到新地址空间）

    线程唯一单例：使用 ThreadLocal 或 ConcurrentHashMap 存放 <线程ID, 实例>

    集群唯一单例：

    • 将单例序列化到外部共享存储区
    • 进程从共享存储读取、反序列化、使用、写回
    • 使用分布式锁保证只有一个进程持有单例

    多例模式

    一个类能创建多个实例，但数量有限

    实现：instances 是静态 HashMap<序号, 实例>，getInstance(long no) 传参获取

    多例 VS 工厂：

    • 多例创建同一类的多个实例
    • 工厂创建不同子类的多个实例

    工厂模式（Factory）

    使用场景：创建对象过程复杂时

    • 设计复杂的 if-else 判断
    • 对象需要组装多个其他类或复杂初始化

    分类：简单工厂、工厂方法、抽象工厂

    依赖注入（DI）框架

    DI 容器 VS 工厂模式：

    • DI 基于工厂模式，在启动时根据配置创建所有对象
    • DI 负责整个应用的对象创建，工厂只负责某一组对象
    • DI 还负责配置解析、生命周期管理

    核心功能：

    1. 配置解析：从配置文件提取创建信息
    2. 对象创建：使用反射动态加载类、创建对象
    3. 生命周期管理：scope(prototype/singleton)、lazy-init、init-method、destroy-method

    建造者模式（Builder）

    使用场景：

    • 构造函数参数过多，容易出错
    • 参数间有制约关系，需要验证
    • 希望创建后对象不可修改

    建造者 VS 工厂：

    • 工厂创建不同但相关的对象
    • 建造者用复杂参数创建一种类型的对象

    原型模式（Prototype）

    适用场景：对象创建成本高，同类对象差别不大时

    实现：深拷贝和浅拷贝

    深拷贝优化：只深拷贝需要更新的部分，其他部分浅拷贝

    第四部分：结构型设计模式

    代理模式（Proxy）

    在不改变原始类的情况下，通过代理类附加功能

    实现方式：

    1. 代理类与原始类实现同一接口
    2. 对外部类，代理类继承原始类

    动态代理：运行时动态创建代理类（Java 的 DynamicProxyHandler）

    应用于：日志、限流、监控、统计、事务、幂等等

    桥接模式（Bridge）

    将抽象和实现解耦，让它们可以独立变化

    例：JDBC 驱动支持不同数据库切换

    装饰器模式（Decorator）

    在不改变原始类接口的情况下，对其功能进行增强，支持多个装饰器嵌套使用

    装饰器 VS 代理：

    • 代理：附加与原始类无关的功能（控制访问）
    • 装饰器：附加与原始类相关的增强功能

    适配器模式（Adapter）

    将不兼容的接口转换为可兼容的接口

    实现方式：

    • 类适配器：继承实现
    • 对象适配器：组合实现

    应用场景：

    • 封装有缺陷的接口设计
    • 统一多个类的接口
    • 替换依赖的外部系统
    • 兼容老版本接口
    • 适配不同格式的数据

    四种包装模式对比

    代理、桥接、装饰器、适配器都属于 Wrapper 模式，代码结构相似但用意不同：

    模式目的客户接口改变应用场景
    代理控制访问同日志、监控、缓存
    装饰器功能增强同增加新功能
    桥接分离抽象实现同支持多种实现切换
    适配器接口转换改变接口不兼容

    门面模式（Facade）

    设计合理的接口粒度以兼顾易用性和通用性

    组合模式（Composite）

    用来处理树形结构数据（一组对象集合）

    享元模式（Flyweight）

    复用对象，节省内存。前提：享元对象是不可变对象

    实现：工厂类通过 Map 缓存享元对象

    享元 VS 其他模式：

    • 享元 VS 单例：单例数量1个，享元多个；目的都是复用
    • 享元 VS 缓存：缓存提高访问速率，享元节省空间
    • 享元 VS 对象池：对象池提高创建速率，享元提高空间效率

    Java 中的应用：

    • Integer -128~127 缓存在 IntegerCache
    • String 字符串常量池
    • 享元对 GC 不友好

    第五部分：行为型设计模式

    观察者模式（Observer）

    也称发布订阅模式。一个对象状态改变，所有观察者都会自动收到通知

    实现方式：

    1. 同步阻塞：观察者直接调用
    2. 异步非阻塞：创建新线程、EventBus、消息队列

    模板模式（Template Method）

    在一个方法中定义算法骨架，将某些步骤推迟到子类实现

    实现：

    • 模板方法定义为 final，不可被子类重写
    • 需要子类重写的方法定义为 abstract

    解决的两个问题：复用、扩展

    模板模式 VS 回调：

    • 回调基于组合，更灵活
    • 模板模式基于继承
    • 同步回调类似模板模式，异步回调类似观察者模式

    策略模式（Strategy）

    定义一组算法，将每个算法封装起来，让它们可以互相替换

    解耦：策略的定义、创建、使用三个部分

    实现：

    1. 策略定义：策略接口和具体策略类
    2. 策略创建：策略工厂类，通过 Map 存储避免 if-else
    3. 策略使用：运行时动态确定使用哪种策略

    职责链模式（Chain of Responsibility）

    将请求和发送解耦，多个接收对象（处理器）链式处理请求

    实现方式：

    1. 链表：handlers 互相指向，head->handler1->handler2->tail
    2. 数组：handlers 存储在 HandlerChain，依次调用

    应用场景：敏感词过滤等，满足开闭原则

    Servlet Filter、Spring Interceptor 都是职责链模式

    状态模式（State）

    有限状态机（FSM）：状态、事件、动作

    实现方式：

    1. 分支逻辑：if-else/switch，简单逻辑可用
    2. 查表法：二维数组 [状态][事件]→新状态，适合复杂状态机
    3. 状态模式：不同状态类，子类重写各自的方法

    迭代器模式（Iterator）

    将集合对象的遍历操作从集合类中拆分到迭代器类

    基本方法：hasNext()、currentItem()、next()

    遍历时增删元素的解决方案：

    • 方案1：不允许增删（难实现）
    • 方案2：增删时抛异常，用 modCount 检测修改

    快照迭代器：

    • 拷贝方案：浅拷贝容器元素
    • 时间戳方案：记录创建时间戳和删除时间戳

    访问者模式（Visitor）

    对象和操作的关注点分离

    Single Dispatch VS Double Dispatch：

    • Single Dispatch：方法由运行时对象类型 + 编译时参数类型决定
    • Double Dispatch：方法由运行时对象类型 + 运行时参数类型决定

    备忘录模式（Memento）

    捕获对象内部状态并保存，以便之后恢复

    内存优化：仅备份前一个操作，或低频全量 + 高频增量备份

    命令模式（Command）

    将请求（命令）封装为对象，支持排队、记录日志、撤销等

    命令 VS 策略：

    • 策略目的相同、实现不同，可互换
    • 命令目的不同、实现不同，不可互换

    解释器模式（Interpreter）

    为某个语言定义语法表示，定义解释器处理这个语法

    中介者模式（Mediator）

    封装一组对象之间的交互，通过中介对象交互而避免直接交互

    例：航空管制

    第六部分：设计模式总结

    何时使用设计模式

    先有问题后有方案：解决实际的代码痛点，不要一开始就引入复杂设计

    设计的应用场景是复杂代码：

    • 项目复杂、周期长、人员多 → 前期投入设计
    • 项目简单、代码少 → 不要引入复杂设计，保持简单

    持续重构能有效避免过度设计：

    • 设计模式提高可扩展性但降低可读性
    • 不确定时，先简单实现，代码量不大时重构成本不高

    避免设计不足

    1. 理论知识储备
    2. 代码训练
    3. 有代码质量意识
      • 写代码前考虑会有哪些扩展需求
      • 预判哪部分会变、哪部分不变

    项目实战：限流框架

    需求背景

    分布式系统调用方多，调用不当或容量瓶颈导致服务资源竞争，解决方案：限流

    需求分析

    • 配置限流规则：YAML 格式
    • 验证请求是否被限流
    • 框架特点：易用性、扩展性、性能、容错

    框架特性需求

    1. 易用性

      • 限流规则、接口简单
      • 支持多种限流算法

    2. 扩展性、灵活性

      • 灵活扩展各种算法
      • 支持多种配置格式（JSON、YAML、XML）

    3. 性能

      • 低延迟响应

    4. 容错

      • Redis 异常不应影响服务本身
      • 限流框架异常不能影响业务正常运行

    限流规则设计

    粒度：足够细粒度才能起到限流作用（如：100/1s，不是 60000/1min）

    配置方式：约定 ./classpath/ratelimiter-rule.yaml

    常见限流算法

    • 固定时间窗口限流
    • 滑动时间窗口限流
    • 令牌桶限流
    • 漏斗限流

    限流模式

    单机限流：每个实例维护自己的计数器

    分布式限流：

    • 所有实例共享计数器（Redis）
    • 异常处理：吞掉异常或向上抛出，继续执行接口
    • 超时处理：合理配置，通过压测获取延迟分布，权衡超时时间

    低侵入集成

    限流框架应低侵入、松耦合，容易替换、删除
